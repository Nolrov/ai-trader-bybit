# AI Trader — Bybit BTCUSDT (15m / 30m)

## Назначение

Система алгоритмической торговли, состоящая из трёх контуров:

- Data Layer — получение, обновление и обработка рыночных данных
- Research Layer — генерация, тестирование и отбор стратегий (alpha)
- Execution Layer — принятие торгового решения, контроль риска и исполнение

Инструмент:
- Bybit
- BTCUSDT (USDT perpetual)

Таймфреймы:
- 15m (основной)
- 30m (старший)

---

## ТЕКУЩИЙ СТАТУС

Система инфраструктурно собрана и работает как единое целое.

Подтверждено:
- единый data layer
- автоматическое обновление данных
- alpha miner работает
- run_candidate работает
- live loop стабилен
- execution работает (paper/testnet)
- risk manager работает
- state синхронизируется
- logging работает

Не готово:
- стабильные стратегии
- достаточное количество сигналов
- банк активных кандидатов
- policy layer

Главный блокер:
**НЕТ СИГНАЛОВ**

То есть система технически работает, но не получает условий для входа в позицию.

---

## КЛЮЧЕВАЯ ИДЕЯ ПРОЕКТА

Это не бот с одной зашитой стратегией.

Целевая схема проекта:

research → банк кандидатов → policy layer → торговое решение → execution

Текущая версия пока переходная:
- используется один кандидат
- кандидат выбирается вручную через `settings.strategy.candidate_id`
- live loop использует этот кандидат как единственный источник сигнала

Это временный текущий механизм. Конечная цель — multi-strategy adaptive system, а не single-strategy execution engine.

---

## DATA LAYER (ЕДИНАЯ ТОЧКА ВХОДА)

Все данные проходят через:

`market_data_manager`

Его контракт:
- проверяет наличие локальных данных
- проверяет freshness
- при необходимости обновляет данные через `bybit_loader`
- читает локальные CSV как кэш
- возвращает готовый dataframe для дальнейшей обработки

Важно:
- CSV — это кэш
- источник истины — биржа
- ручное обновление данных отсутствует
- ни один модуль не должен напрямую читать CSV или напрямую вызывать `bybit_loader` для торговой логики

Жёсткий архитектурный инвариант:
**все модули используют только `market_data_manager` как вход в market data pipeline**

## КАК ПРИНИМАЕТСЯ РЕШЕНИЕ

В текущей версии система работает так:

1. Загружается market data через `market_data_manager`
2. Строится обработанный dataframe
3. Выбранный кандидат применяется через `apply_candidate`
4. Стратегия генерирует:
   - `entry_signal`
   - `position`
5. `position` интерпретируется как целевое состояние позиции (`desired_position`)
6. `risk_manager` решает, разрешать ли изменение позиции
7. Только после этого возможен ордер

То есть логика не такая:

signal → order

А такая:

signal → desired_position → risk → execution → order

---

## ЛОГИКА СИГНАЛА

Стратегия генерирует:

- `entry_signal ∈ {-1, 0, 1}`
- `position ∈ {-1, 0, 1}`

Смысл:
- `entry_signal` — факт входного сигнала
- `position` — желаемое удержание позиции по логике стратегии
- `desired_position` — то, что затем уходит в risk manager

Текущая упрощённая модель:
- стратегия даёт вход через `entry_signal`
- удержание строится через position logic (hold-based)
- `position` используется как торговое намерение

### Поведение при отсутствии сигнала

Если:
- `entry_signal = 0`
- `desired_position = 0`

то система ничего не делает.

Это нормальное штатное поведение.
Отсутствие сделки не считается ошибкой.

---

## APPLY_CANDIDATE

`apply_candidate` — это не абстрактный шаг, а фактическое применение выбранной стратегии к dataframe.

Он делает две вещи:
- рассчитывает стратегический сигнал
- строит позицию по параметрам кандидата

То есть это точка, где параметры кандидата становятся реальным торговым поведением.

---

## RISK MANAGER

Risk manager — обязательный фильтр перед execution.

Его задача:
- принять `desired_position`
- сравнить с текущим состоянием
- либо разрешить изменение
- либо запретить изменение

Risk manager:
- не генерирует сигнал
- не выбирает стратегию
- не принимает alpha-решение

Он только контролирует, можно ли передать решение дальше в execution.

Жёсткий инвариант:
**любое изменение позиции проходит через risk manager**

## STATE

Файл:

`data/live_state.json`

Содержит:
- текущую позицию
- размер позиции
- цену входа
- служебные поля
- snapshot состояния биржи

Важно:
- state не является источником истины
- state синхронизируется с биржей каждый цикл
- источник истины по позиции — биржа

## ПОВЕДЕНИЕ ПРИ ОШИБКАХ

Система должна вести себя консервативно.

Если:
- exchange sync неуспешен
- market data не получены
- данные stale
- обработанный dataframe пуст
- execution не подтверждён

то торговое действие не должно выполняться.

Жёсткий инвариант:
**без успешного exchange_sync торговля запрещена**

## ТЕКУЩЕЕ ОГРАНИЧЕНИЕ

Сейчас система:
- использует один `candidate_id`
- не умеет автоматически выбирать стратегию
- не умеет переключать стратегии
- не имеет policy layer
- не адаптируется к рынку на уровне управления банком кандидатов

То есть сейчас это:
**single-strategy execution engine**

А должно стать:
**multi-strategy adaptive trading system**

---

## ИНВАРИАНТЫ СИСТЕМЫ

- данные всегда обновляются автоматически
- все модули используют один data source
- ни один модуль не должен напрямую читать market CSV для торговой логики
- без успешного exchange_sync торговля запрещена
- risk_manager всегда применяется перед execution
- отсутствие сигнала не является ошибкой
- биржа — источник истины по позиции
- локальный state — производный, а не первичный

## СЛЕДУЮЩИЙ ЭТАП

1. Банк кандидатов (`active_candidates`)
2. Policy Manager:
   - выбор стратегии
   - выбор позиции
   - решение “торговать / не торговать”
3. Удаление ручного `candidate_id` из `settings`
